How it works

Three ideas, no surprises.

Subtick is intentionally small. The transactions are real, the signatures are real, and the chain executes them as fast as it can — every tick, not every block.

Real-time

~130 ms from POST to executed. No block window, no batching delay — every signed tx runs the moment the chain sees it.

Signed & strict

Ed25519 signatures, strict nonces, real on-chain state. Not a mock, not a queue — replay, double-spend, and stale txs are rejected.

Live

WebSocket events stream as the chain commits each batch. Subscribe and react — no polling, no reconciliation, no stale views.

Live

A pulse from the chain.

Streaming straight from wss://subtick.dev/v1/events. Each row is a batch the chain just executed.

Quickstart

Talk to the chain in five lines.

The SDK is a thin wrapper around the HTTP/WS API. Bring your own crypto, or use the helpers.

example.js 
import { SubtickClient } from '@subtick/sdk';

const c = new SubtickClient({ baseUrl: 'https://subtick.dev' });

const health = await c.health();
console.log(health);
// { status: 'ok', height: 8421023, slot: ..., account_count: ... }
API

A tiny, honest surface.

No auth, no rate limits, no historical queries — yet. The v0 contract is locked and deliberately small.

Method Path Use
GET /health liveness + chain status
POST/v1/tx submit a signed transaction
GET /v1/balance/:addr balance for an address
GET /v1/account/:addr balance + nonce
WS /v1/events live execution stream
